# Synthetic Patient Dataset

> ⚠️ **100% SYNTHETIC DATA. NOT REAL PATIENTS.**
> Every record here is fabricated for a portfolio simulation. No real person, NHS number,
> address, or phone number is represented. Do not treat any value as real PII.

## How "synthetic" is guaranteed

| Field | Synthetic-by-construction rule |
|---|---|
| NHS number | Reserved **999 test range** (`999xxxxxxx`) with a **valid Modulus-11 check digit**. The 999-prefix range is reserved by NHS Digital for test/synthetic use and is never issued to a real patient. |
| Phone | **Ofcom reserved fictional range `07700 900000–900999`** — guaranteed never assigned to a real subscriber. |
| GP practice | Fictional practice names. |
| Names / DOB / postcode | Fabricated. Postcodes are plausible-format but not tied to any individual. |

## Files

- `synthetic_patients.json` — full dataset (source of truth; consumed by the seeder and later by referral generation / matching tests).
- `synthetic_patients.csv` — same data, spreadsheet-friendly for screenshots.

Both are generated by [`../../openmrs-setup/seed-data/generate_patients.py`](../../openmrs-setup/seed-data/generate_patients.py)
and are deterministic (re-running produces identical data).

## Scenario design (drives Phase 5 patient-matching tests)

The dataset is curated, not random, so later phases have concrete fixtures:

| `scenario` tag | Purpose | `seed` |
|---|---|---|
| `standard` | Exact-match pool — referral matches on NHS + DOB | true |
| `dob_mismatch_target` | Referral will carry this NHS but a wrong DOB → human review | true |
| `partial_match_target` | Referral omits NHS; match on surname + DOB + postcode → human review | true |
| `multiple_candidate` | Two patients share surname + DOB (different NHS/postcode) → ambiguous demographic search → human review | true |
| `duplicate_target` | Has an existing/active referral; a new referral in the same speciality → duplicate review. (The pre-existing referral encounter is seeded in Phase 6/8 once the `Referral` encounter type + concepts exist.) | true |
| `no_match` | Identity referenced by a referral but **deliberately NOT loaded** into OpenMRS → business exception / human review | **false** |

## What is loaded into OpenMRS vs kept here

- **Loaded into OpenMRS** (needed for matching): NHS number (as `Synthetic NHS Number`
  identifier), given/family name, gender, birthdate, postcode (address), phone (Telephone
  Number attribute).
- **Kept in this dataset only** (used by referral generation / docs, not required for
  matching): GP practice, conditions/alerts, existing-referral status. These can be added to
  OpenMRS later as conditions/attributes if a phase needs them.
